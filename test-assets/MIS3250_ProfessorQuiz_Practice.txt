Section B:
13. Atomicity: all operations complete or none do (all-or-nothing).
    Consistency: transaction moves DB from one valid state to another, respecting all constraints.
    Isolation: concurrent transactions behave as if they execute serially; intermediate states invisible.
    Durability: committed changes are permanent even after system failure (via WAL/checkpoints).

14. FDs:
    - ProductID → ProductName, ProductPrice  (partial dependency on part of PK)
    - CustomerID → CustomerName              (non-key → non-key, via OrderID→CustomerID)
    - OrderID → CustomerID                   (partial dependency on part of PK)
    - OrderID, ProductID → Quantity          (full dependency — correct)

    Current form: 1NF only (has partial dependencies → violates 2NF)

    3NF decomposition:
    - Product(ProductID, ProductName, ProductPrice)
    - Customer(CustomerID, CustomerName)
    - Order(OrderID, CustomerID)
    - OrderItem(OrderID, ProductID, Quantity)

15. A non-correlated subquery runs once and returns a result used by the outer query.
    A correlated subquery references a column from the outer query and re-executes for every row.

    Prefer JOINs over correlated subqueries when: the operation can be expressed as a join (which
    the optimizer can better parallelize and index-seek), and when performance matters on large
    tables — correlated subqueries are O(n) executions.

16. a) Composite index on (DeptID, Salary) — covers both filter columns in one index scan.
        Put DeptID first (equality filter) then Salary (range filter): idx on (DeptID, Salary).
    b) No — Hash index cannot handle range conditions (Salary > 80000). Hash indexes only
       support equality (=). A B+ tree is needed for range queries.
    c) Selectivity = number of distinct values / total rows. High selectivity → index is efficient
       because it narrows down rows quickly. DeptID alone may have low selectivity (few depts),
       but the composite (DeptID, Salary) with the range filter will be highly selective.

17. States: Active → Partially Committed → Committed
                                        ↘ Failed → Aborted
            Active → Failed → Aborted

    "Partially Committed" to "Failed": occurs when post-execution checks fail — e.g., a constraint
    violation is detected during pre-commit validation, or the system cannot write to disk (I/O error).

Section C:

18. SELECT s.Name, s.GPA
    FROM Student s
    WHERE s.GPA > (
      SELECT AVG(s2.GPA)
      FROM Student s2
      WHERE s2.DeptID = s.DeptID
    )
    ORDER BY s.GPA DESC;

19. WITH EnrolledStudents AS (
      SELECT s.StudentID, s.DeptID, s.GPA
      FROM Student s
      JOIN Enrollment e ON s.StudentID = e.StudentID
    ),
    DeptStats AS (
      SELECT d.DeptName,
             COUNT(DISTINCT es.StudentID) AS NumStudents,
             AVG(es.GPA) AS AvgGPA
      FROM Department d
      JOIN EnrolledStudents es ON d.DeptID = es.DeptID
      GROUP BY d.DeptID, d.DeptName
      HAVING COUNT(DISTINCT es.StudentID) > 100
    )
    SELECT DeptName, NumStudents, ROUND(AvgGPA, 2) AS AvgGPA
    FROM DeptStats
    ORDER BY NumStudents DESC;

20. SELECT
      StudentID,
      Name,
      DeptID,
      GPA,
      RANK() OVER (PARTITION BY DeptID ORDER BY GPA DESC) AS RankInDept,
      ROUND(GPA - AVG(GPA) OVER (PARTITION BY DeptID), 2) AS GPAVsDeptAvg
    FROM Student
    ORDER BY DeptID, RankInDept;
